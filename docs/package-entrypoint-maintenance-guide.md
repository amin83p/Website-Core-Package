# Package Entry Point Maintenance Guide

This guide explains how to create and maintain a package entrypoint so package source files, pages, routes, views, and assets stay reachable without manually listing every internal file in the manifest.

Use this whenever a package gains new pages, child route files, upload categories, public assets, or package-level runtime wiring.

## Reusable Model Prompt

Replace only `<PACKAGE_ID>` with the package folder name, for example `school`, `pte`, `ielts`, `benchpath`, or `credit`.

```text
Package id/name: <PACKAGE_ID>

Please audit and update the package entrypoint for <PACKAGE_ID>.

Use the package-owned structure under packages/<PACKAGE_ID>. Inspect:
- packages/<PACKAGE_ID>/package.manifest.json
- packages/<PACKAGE_ID>/MVC/routes
- packages/<PACKAGE_ID>/MVC/controllers
- packages/<PACKAGE_ID>/MVC/views
- packages/<PACKAGE_ID>/public, if present
- package config/access constants, if present

Goal:
Make sure the package entrypoint is complete and current. The manifest should keep package-level runtime wiring only, while the main route should mount all package-local child route files required by current features.

Rules:
- Keep one active runtime USE route in the manifest for the package mount path.
- Keep views/assets roots package-local.
- Do not list every internal page/controller/view in manifest routes.
- Add or update child route mounts in the main router when package route files exist and should be reachable.
- Keep imports package-local or through approved core resolver/dependency adapters.
- Preserve live manifest behavior for business declarations.
- Report anything that cannot be inferred safely.
```

## Entry Point Contract

A separated package should have one clear runtime entrypoint in its manifest and one main router file inside the package.

The manifest should include:

- `routes`: one active runtime `USE` declaration for the package mount path, for example `/pte`, `/school`, or `/credit`.
- `views`: one package-local view root, normally `packages/<packageId>/MVC/views`.
- `assets`: one package-local public/assets root when the package has public assets.
- Other package-level wiring only when needed, such as `uploadFolders`, `mongoIndexes`, `migrations`, `seeders`, `dataSchemas`, `upgradeGuards`, or `dependencies`.

The main route file should:

- Export an Express router.
- Mount package-local child route files with `router.use(...)`.
- Register package upload category resolvers when the package has upload-specific categories.
- Set package-specific `res.locals` values when dashboards or views need them.
- Keep route/controller/service imports package-local or routed through approved core resolver/dependency adapters.

## Update Rules

Update the package entrypoint when:

- A new child route file is added under `packages/<packageId>/MVC/routes`.
- A new package root route is added, such as `/`, `/dashboard`, `/join`, or another public package route.
- A new route tree should become reachable under the package mount path.
- A new upload category registration hook is required.
- A new package public/assets root is added.
- The package manifest no longer points to the correct main router, view root, or asset root.

Do not update manifest route declarations for ordinary internal additions when they are already inside the mounted package route tree:

- New EJS files under the existing package view root.
- New controllers, services, models, or repositories used by existing mounted route files.
- New endpoints added inside an already-mounted child route file.
- New package data tables handled by package data services and live manifest generation.
- New sections, roles, operations, symbols, menu entries, dashboard entries, or quota toggles that live manifest generation can detect.

## Live Manifest Boundary

The entrypoint handles code and runtime wiring. Live manifest generation handles business declarations.

Preserve these from the package template unless the package capability changed:

- `routes`
- `views`
- `assets`
- `uploadFolders`
- `mongoIndexes`
- `seeders`
- `migrations`
- `dataSchemas`
- `upgradeGuards`
- `dependencies`

Let live manifest generation derive or refresh these where supported:

- `sections`
- `roles`
- `operations`
- `symbols`
- `menuEntries`
- `dashboardEntries`
- `quotaDefinitions`
- `dataEntities`

Access profiles are usually not generated by default because different app instances may assign users and admins differently.

## Entry Point Patterns

Use PTE as the rich pattern when a package has public pages, dashboard links, upload category registration, and many child route files.

Use School as the broad route-tree pattern when a package mostly mounts many internal feature route files under one package root.

Use IELTS as the compact bridge pattern when the package has one main route tree and needs a small amount of shared core resolver setup.

Use Credit as the minimum acceptable pattern when the package currently only needs one package-local child route tree.

## Validation Checklist

After updating a package entrypoint, confirm:

- The manifest validates with the strict package manifest service.
- The package loader route report shows `mounted=1` and `failed=0` for the runtime `USE` route.
- The package root route, for example `/pte` or `/school`, is reachable.
- Any newly added page route is reachable.
- Manifest routes do not manually list every internal page/controller/view.
- The manifest `views` root points to the package view directory.
- The manifest `assets` root points to the package public directory when assets exist.
- There are no root `MVC/*/<packageId>` runtime ownership folders left for fully separated packages.
- Live manifest preflight still captures expected sections, roles, operations, symbols, quota toggles, and data entities.

## Model Output Expectations

When using the reusable prompt, the model should return:

- A short summary of entrypoint changes made.
- A list of child route files mounted by the main router.
- Any package-local route files that were not mounted, with a reason.
- Any missing view/assets roots.
- Any root MVC leftovers that conflict with full package ownership.
- Any live manifest declaration risks, such as roles or operations that lack package markers.
